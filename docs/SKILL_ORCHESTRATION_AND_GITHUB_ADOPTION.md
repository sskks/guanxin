# Skill 编排与执行标准

> 最后更新：2026-06-30
> 适用范围：`观心` 产品、UI、前端与文档协作

## 1. 目的

本文件用于约束后续开发中各类 skill 的职责边界，避免出现：

- 只改前端视觉，不校准产品方向
- 只改文案，不校准页面层级
- 只加功能，不校准 MVP 阶段目标
- 多个 skill 同时介入但输出互相冲突

## 2. 当前项目阶段判断

当前项目处于：

`Development + Testing` 之间的 `MVP 主路径收口阶段`

因此 skill 的使用优先级不是“横向扩功能”，而是：

1. 先对齐产品阶段
2. 再对齐页面层级
3. 再做 UI / 前端实现
4. 最后做可视验证与文档同步

## 3. Skill 职责分工

### `product-generation-ops`

负责：
- 判断当前所处产品阶段
- 控制本轮 scope
- 决定是做 P0 收口还是进入 P1/P2

### `product-manager-toolkit`

负责：
- 结构化拆任务
- 输出功能优先级
- 形成页面职责与验收口径

### `ui-ux-pro-max`

负责：
- 信息层级
- 移动端可读性
- 导航、返回、模态、可触达性
- 交互显隐顺序

### `frontend-design`

负责：
- 把既定结构落实成更完整的前端视觉实现
- 避免通用模板感

### `impeccable` / `design-taste-frontend`

负责：
- 识别“AI 味”页面
- 防止大卡片堆叠、同构布局、廉价视觉

### `playwright` / `screenshot`

负责：
- 渲染结果验证
- 移动端可视化检查

## 4. 标准执行顺序

每次较大改动必须按以下顺序执行：

1. `product-generation-ops`
2. `product-manager-toolkit`
3. `ui-ux-pro-max`
4. `frontend-design`
5. `playwright / screenshot`

## 5. 观心项目专用判断规则

### 不能直接改 UI 的情况

1. 页面职责说不清
2. 页面主次路径冲突
3. 一个页面承担了多个模块职责
4. 当前改动可能让首页重新变重

### 不能直接扩功能的情况

1. 首页层级仍然模糊
2. 用户分不清“这是哪个模块”
3. 返回路径不稳定
4. 文本仍明显重复
5. 移动端滚动与弹层仍不顺

### 进入 P1 的前提

1. 首页主次路径稳定
2. 看自己页四层结构清楚
3. 问一事结果页去重完成
4. AI 触发与追问逻辑稳定
5. 历史 / 收藏 / 回看基本可用

## 6. 与 GitHub 的协同规则

当前仓库：`sskks/guanxin`

规则：

1. GitHub 主分支只作为“最近稳定基线”
2. 本地改动前先以文档确认当前目标
3. 每次推送前同步更新：
   - `docs/PROJECT_STATE.md`
   - `docs/DEVELOPMENT_PLAN.md`
   - 与本轮相关的规格文档
4. 未经过渲染验收的内容，不写成“已完成”

## 7. 当前恢复后的执行结论

本轮从 GitHub 恢复仓库后，当前优先级为：

### P0

1. 首页首屏层级继续收口
2. 命宫档案页减重并强化四层切换
3. 问一事页结果首屏去重
4. AI 追问区只在触发后出现

### 暂不做

1. 新术数扩展
2. 微社交
3. 新内容大扩写
4. 大规模架构拆分

## 8. 使用原则总结

一句话标准：

`先定方向，再定结构，再做视觉，再验渲染，再更新文档。`
