# 商业化架构演进方案

> 最后更新：2026-06-22

---

## 1. 结论

观心后续要走商业化，必须从当前“纯前端单文件 MVP”逐步演进为“前端应用 + 后端服务 + 数据与权益系统”的结构。

当前不建议立刻重写为大型框架项目。更稳妥的路线是：

1. 保持 `index.html` 可运行
2. 先建立前后端目录边界
3. 抽离设计系统、领域逻辑和内容数据
4. 再引入账号、同步、支付、内容管理

---

## 2. 目标架构

```text
Cculture/
├── apps/
│   ├── web/              # 前端应用
│   └── api/              # 后端 API
├── packages/
│   ├── design-system/    # 设计 token 与通用组件样式
│   ├── domain/           # 起卦、命宫、八宅、主题分类等纯逻辑
│   └── content/          # 卦象解释、术语、prompt、运营内容
├── docs/                 # 产品、架构、开发与商业化文档
├── index.html            # 当前线上入口，迁移期继续保留
└── server.py             # 当前本地服务器与 AI 代理
```

---

## 3. 前端演进

### 当前阶段

- 单文件 `index.html`
- 原生 HTML / CSS / JavaScript
- 无构建工具
- 本地存储依赖 `localStorage` 和 `IndexedDB`

### 过渡阶段

- 抽离 `:root` token 到 `packages/design-system/tokens.css`
- 抽离通用组件样式到 `packages/design-system/components.css`
- 抽离纯数据到 `packages/domain` 或 `packages/content`
- 建立页面级结构，但保持 `index.html` 可运行

### 商业化阶段

建议再评估是否引入现代前端框架。触发条件：

- 需要账号体系和跨端同步
- 需要支付、会员权益、内容管理后台
- 页面状态和路由复杂度明显提升
- 需要更稳定的自动化测试与发布流水线

---

## 4. 后端演进

### 当前阶段

`server.py` 提供：

- 静态文件服务
- `/api/ai-interpret` AI 代理
- 从环境变量或请求头读取 DashScope Key

### 第一阶段后端

- 统一 AI 代理响应格式
- 增加错误码和错误恢复提示
- 增加基础请求日志
- 增加模型配置和最大 token 限制
- 保留本地 Key 直连兜底能力

### 第二阶段后端

- 匿名用户 ID
- 用户登录
- 记录同步
- 收藏同步
- 回看提醒
- 设备间数据迁移

### 第三阶段后端

- 订阅和一次性付费
- AI 解读额度
- 会员内容
- 支付 Webhook
- 运营后台
- 内容审核与风控

---

## 5. 数据分层

### 本地数据

适合继续放本地：

- 未登录用户出生信息
- 本地占卦历史
- 本地收藏
- 新手引导状态
- 临时 Qwen Key

### 服务端数据

商业化后应迁移到服务端：

- 登录用户资料
- 跨设备占卦记录
- 收藏与回看状态
- 会员权益
- 订单和支付状态
- AI 使用额度
- 运营内容

---

## 6. 商业化能力清单

### 可免费

- 每日一卦
- 基础图鉴
- 基础问卦
- 本地历史

### 可会员

- 深度 AI 解读
- 月度回顾
- 主题档案
- 跨设备同步
- 节气专题
- 个性化成长建议

### 可单次付费

- 深度报告
- 年度回顾
- 关系 / 事业 / 情绪专题报告

---

## 7. 风险

- 过早引入账号和支付会拖慢 MVP 收口
- 纯前端继续堆功能会造成维护风险
- AI 成本需要在商业化前建立额度和限流
- 情绪类产品必须明确边界，避免制造依赖或替用户决策

---

## 8. 推荐执行顺序

1. 保持当前 MVP 可运行
2. 建立 `PRODUCT.md` / `DESIGN.md`
3. 建立 `apps/` 和 `packages/` 目录边界
4. 抽离设计 token
5. 抽离领域逻辑
6. 抽离内容与 prompt
7. 统一 AI 代理接口
8. 再评估账号、数据库、支付和现代前端框架
